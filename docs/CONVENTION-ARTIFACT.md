# The Convention artifact (procedural memory)

Renamed from "CLAUDE.md edit" (2026-06-30). The old name was wrong twice: the artifact
never touches the user's `CLAUDE.md`, and it is not "an edit" but a learned, reviewed
rule. `ArtifactType.CONVENTION` keeps a `_missing_` shim mapping the legacy `claude_md`
value so old catalog cards still load.

## 1. What it is

A **convention** is the SOFT sibling of a **Rule**: a learned rule / style / preference
we agreed on while working with the agent, that cannot be a deterministic gate, so it is
delivered as context to steer the model rather than a hook to block it.

In agent-memory terms this is **procedural memory** (learned behaviors and rules), as
opposed to **semantic** memory (facts, the Knowledge artifact) or **episodic** memory
(past interactions). The distinction matters for retrieval (see §6): a convention is
*applied proactively*, not *queried*, so its relevance signal is the working context
(which files/tools/activity are in play), not similarity to a question.

It rides the exact same pipeline as a Rule (`DETECT → REVIEW → COMPILE → COMMIT`). Only
the last step differs: a Rule's COMMIT writes a blocking policy into the hook cache; a
convention's COMMIT writes plain text into a file Claude Code loads as context.

## 2. Where it writes, and CLAUDE.md vs .claude/rules

Conventions are written to Precept-OWNED files under `.claude/rules/`, **never** into the
user's `CLAUDE.md`. The two are different mechanisms (both delivered as a user-message
context block after the system prompt, neither is the system prompt itself):

| | `CLAUDE.md` (the file) | `.claude/rules/*.md` (a folder) |
|---|---|---|
| count | one file per level | many files in a folder |
| when loaded | always, in full | no `paths:` → always; `paths:` → only on a matching file |
| scoping | by directory nesting only | per-file `paths:` glob |
| authored by | the user, by hand | user OR tools (docs recommend generated content here) |
| Precept writes? | no (never clobber the user's memory) | yes (owns its own files) |

Scope (set at DETECT, stored on the `Lesson`) picks the file, via `convention.target_for`:

| scope | file | loads |
|---|---|---|
| `GLOBAL` | `~/.claude/rules/precept-conventions.md` | every session |
| `REPO` (`scope_value`=root) | `<root>/.claude/rules/precept-conventions.md` | sessions in that repo |
| `LANGUAGE` (`scope_value`=marker) | `~/.claude/rules/precept-<marker>.md` | only when a matching file is touched (`paths:` glob) |

Why owned files, not splicing into `CLAUDE.md`: the Claude Code docs (verified
`code.claude.com/docs/en/memory`, 2026-06-30) do not endorse third-party in-place edits
and recommend a decoupled file via the native `.claude/rules/` directory. Owning the whole
file makes updates "regenerate" and uninstall "delete", with zero clobber risk and no
marker bookkeeping. A sidecar manifest (`state_dir/managed_conventions.json`) records
Precept's own files so cleanup removes only ours.

## 3. File structure

The file is a pure function of the active kept conventions (`convention.render_file`),
regenerated wholesale each compile, sorted by lesson id, written atomically. A GLOBAL/REPO
file:

```
<!-- Generated by Precept — do not edit. Managed via `precept keep` / `precept delete`. -->
# Conventions (managed by Precept)

- document every public function
- prefer composition over inheritance
```

A LANGUAGE file is identical except for leading `paths:` frontmatter, which is what makes
Claude Code lazy-load it:

```
---
paths:
  - "**/*.py"
---
<!-- Generated by Precept — do not edit. ... -->
# Conventions (managed by Precept)

- always add type hints
```

Each bullet is one lesson's `what_to_do_instead`.

## 4. Flow, and the honesty boundary

Authoring (Precept): a correction → DETECT (Haiku) mints a PENDING card →
`review.py` surfaces it → `precept keep` → `compile_all` → `convention.write_managed_rules`
atomically writes the file + reconciles the manifest. Precept's involvement ends at the
file.

Retrieval (Claude Code native, no Precept code): at session start Claude Code reads
`.claude/rules/`; unscoped files inject immediately, `paths:` files inject when a matching
file is read; the bullets become tokens in the context window.

| stage | who | promised? |
|---|---|---|
| correct file at the right path, = active catalog, atomic, idempotent | Precept | **yes**, deterministic |
| only Precept's own files added/removed | Precept | **yes** (manifest) |
| unscoped file loaded at session start | Claude Code | by its documented contract; Precept does not run it |
| `paths:` file loaded | Claude Code | conditional: only if a matching file is touched |
| the model obeys the convention | the model | **no** — soft, probabilistic |

Precept guarantees the artifact exists and is correct. Loading and obedience are out of
its control, which is the definition of SOFT and the reason anything that must be
guaranteed has to be a Rule (hook), not a convention.

## 5. Boundary: only write back what we LEARNED

A convention with `origin=BOOTSTRAP` was imported FROM the user's own `CLAUDE.md`
(`bootstrap.import_claude_md`). Those are NOT re-emitted (`convention.is_managed` filters
them out), because the directive already lives in the user's file and re-emitting would
duplicate it in context. This mirrors how Precept never re-adopts the user's pre-existing
permission entries.

## 6. Design rationale: relevance, and why not a vector DB by default

The hard question for a SOFT artifact is **which conventions enter context, when**.
Best-practice sources converge on "not all of them, always":

- Anthropic context-engineering: "fill the context window with just the right information
  for the next step"; context is finite, adherence degrades as it fills.
- Claude Code memory docs: keep memory lean (rules drop past ~80 lines); use `paths:`
  path-scoping so instructions load only when relevant.
- Karpathy's retrieval pattern is **index-first drill-down**, not inject-all.
- Letta/MemGPT keeps **procedural memory in a small always-in-context core** and reserves
  the **vector store for archival facts** — it does NOT vector-retrieve behavioral rules.

So the relevance signal for a convention is mostly **structural** (the file/tool/activity
in play), which deterministic matching + `paths:` globs already provide, auditably. A
vector DB is the wrong default because:

1. it answers "semantically similar to a query" — but a convention applies to an *action*,
   not a question (relevance is conditional, not similarity);
2. the set is small (tens–hundreds), so it has no scale need;
3. auditability — Precept's ethos is explainable matching; a cosine score is not auditable;
4. terse, jargon-dense rule cards embed poorly (see DECISIONS.md: keyword-first).

A local vector layer is justified ONLY for the genuinely fuzzy subset ("prefer
composition") where applicability really is semantic relevance to the current code, and
ONLY if a Recall@k eval proves keyword/structural retrieval misses, and ONLY local (rules
never leave the machine). Secondary, not default.

## 7. Known limitations / seams

- **The GLOBAL file is always-loaded.** `render_file` writes every active GLOBAL
  convention into one unscoped file, so it spends context every session and will, unbounded,
  trip the degradation the docs warn about. Needs a leanness discipline.
- **No relevance gating beyond `paths:`.** All conventions in a scope bucket load together;
  there is no per-convention relevance ranking yet.
- **`is_managed` does not exclude lessons that also got a hard policy.** A convention that
  the classifier marked `judgment` could be double-implemented (a hard Stop gate AND a soft
  rules-file line). One-line guard: `and not lesson.policies`.
- **Overlap with native auto-memory.** Claude Code's `MEMORY.md` (notes it self-writes from
  corrections) is the same space; Precept's edge is the human keep/veto gate, the typed
  cross-tool catalog, scoping, and that the same pipeline can compile a HARD rule.

## 8. Roadmap

- **P0 (shipped).** Owned `.claude/rules/` files; GLOBAL/REPO/LANGUAGE scope; `paths:`
  path-scoping; manifest cleanup; atomic writes; bootstrap boundary; rename to CONVENTION.
- **P1 (shipped).** (a) `is_managed` excludes a convention that also compiled to a HARD
  policy (no double-implementation). (b) Leanness guardrail: `oversize_files` + a `precept
  doctor` warning past `MAX_RECOMMENDED_LINES`. (c) **Activity-keyed retrieval.** A
  `Lesson.retrieval_only` flag (default False) moves a convention OFF the always-on file and
  into a just-in-time path: `convention.relevant(prompt, cwd)` returns the retrieval_only
  conventions whose scope matches cwd AND whose keywords overlap the prompt (structural token
  match, not semantic); `convention.retrieval_context` formats them; `enforce.evaluate_
  userpromptsubmit` injects them as `additionalContext` alongside knowledge retrieval (lazy,
  fail-open). Always-on and retrieval_only are mutually exclusive, so a convention is never
  double-served. Native `paths:` scoping already covers FILE-keyed relevance, so this targets
  PROMPT/TASK-keyed conventions that path globs can't reach.
  - *Deferred within P1:* AUTO-routing — deciding WHICH conventions become `retrieval_only`
    (vs always-on core) is still manual / a future router job. The flag + mechanism ship now;
    nothing is auto-flagged, so default behavior is unchanged.
- **P2 — fuzzy-subset vector layer (eval-gated).** A LOCAL vector index over fuzzy
  conventions, keyed on the working context (current file/diff), behind a Recall@k eval that
  proves keyword retrieval misses. Off by default. (Keyword retrieval has poor recall on
  fuzzy conventions like "prefer composition" whose trigger words rarely appear in the
  prompt — exactly the subset vectors would help, and the reason auto-routing must keep
  fuzzy/triggerless conventions in the always-on core rather than retrieval_only.)

The end-state is Letta-shaped: a tiny always-in-context core of universal/fuzzy rules, the
keyword-triggerable long tail injected just-in-time by relevance, vectors only where measured
to help.
